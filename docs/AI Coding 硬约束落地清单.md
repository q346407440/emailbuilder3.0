# AI Coding 硬约束落地清单

## 1. 目的

本清单用于将《源头驱动的契约式开发》从理念转化为可执行的工程约束。

它回答的问题不是“AI 应该如何理解需求”，而是：

- 项目中哪些地方必须建立硬约束。
- 哪些约束应优先落到代码和自动化体系中。
- 哪些检查能够阻止 AI 或人工开发中的漏改问题穿透到运行阶段。

本清单适用于新项目初始化，也适用于存量项目的架构治理和迭代整改。

## 2. 使用方式

建议在以下场景使用本清单：

- 新项目立项时，作为架构设计检查表。
- 存量项目重构时，作为约束补齐清单。
- AI Coding 场景下，作为项目工程底座建设清单。
- 需求较大、影响链路较长时，作为改造优先级清单。

推荐使用方式：

1. 先识别系统中的关键源头定义。
2. 再对照本清单确认哪些硬约束已存在、哪些缺失。
3. 对缺失项按优先级补齐。
4. 将清单中的关键项接入 CI、测试和发布流程。

## 3. 硬约束优先级

建议按照以下优先级推进：

### P0：必须具备

这些约束直接决定系统是否具备“阻止不完整修改”的基本能力：

- 单一源头定义
- 类型检查或结构校验
- 模块边界契约
- 基础自动化测试
- CI 阻断能力

### P1：强烈建议具备

这些约束决定系统是否能在中长期迭代中保持稳定：

- 自动派生产物
- 穷尽检查
- 契约测试
- 集成测试
- 发布门禁

### P2：高复杂系统应具备

这些约束对多系统协作、异步场景和外部依赖尤为重要：

- 迁移验证
- 回放测试
- 沙箱校验
- 运行时保护
- 灰度和回滚机制

## 4. 源头层检查项

### 4.1 是否存在明确的源头定义

检查项：

- 接口结构是否有统一契约定义。
- 数据结构是否有统一 schema 或 migration 定义。
- 业务状态是否有统一状态模型或状态机定义。
- 事件格式是否有统一 event schema。
- 权限规则是否有统一 permission model。
- 三方对接是否有统一 adapter contract。

通过标准：

- 同一业务事实只在一个地方被定义。
- 下游模块不再手写等价结构。

失败信号：

- 同一个字段在多个模块中重复定义。
- 同一个状态在多个服务或页面中以不同常量表达。
- 三方回调字段在不同处理器里被各自解释。

### 4.2 是否建立“先改源头”的默认路径

检查项：

- 是否有明确规范要求需求变更先进入源头层。
- 是否有目录结构或模块边界帮助开发者快速定位源头。
- 是否有 review 规则检查是否绕开了源头直接改消费层。

通过标准：

- 团队能快速回答“这次改动的源头定义是什么”。
- 没有“直接改页面、直接改 handler、直接改 mapper”的习惯性路径。

## 5. 派生层检查项

### 5.1 是否尽量由源头自动派生

检查项：

- 类型是否从契约、schema、定义文件自动生成。
- SDK 是否从接口契约自动生成。
- 表单、配置面板、字段校验是否可从 schema 派生。
- 数据映射是否可以从统一结构导出。
- mock 和示例数据是否围绕契约生成或校验。

通过标准：

- 重复结构以“生成”或“统一派生”为主，而不是人工双写。

失败信号：

- 前端、后端、文档、测试中分别手写一份字段结构。
- 一次结构变更要手动同步修改多个模块中的相似代码。

### 5.2 派生产物是否被自动校验

检查项：

- 生成结果是否纳入 CI 检查。
- 是否能发现产物过期、未更新或不一致。
- 是否能在本地和 CI 中重复生成相同结果。

通过标准：

- “源头改了但生成结果没更新”能够被自动阻止。

## 6. 编译与静态校验检查项

### 6.1 类型系统是否表达关键业务约束

检查项：

- 是否使用明确的类型、枚举、联合类型表达关键状态。
- 是否避免随意字符串、任意对象、宽松字典污染核心逻辑。
- 是否在边界处对外部输入做 schema 校验。

通过标准：

- 常见结构变更能触发编译或类型失败。

失败信号：

- 改了字段名但大量调用方无感知。
- 改了状态值但分支逻辑不报错。

### 6.2 是否存在穷尽检查

检查项：

- 关键 switch / match / dispatcher 是否做了 exhaustiveness check。
- 新增状态、新增事件、新增组件类型时，未补全处理逻辑是否会失败。

通过标准：

- 新增分支未处理时，编译器或静态检查器能直接报错。

### 6.3 是否存在自定义静态规则

检查项：

- 是否有 lint rule 阻止绕过源头直接访问底层细节。
- 是否限制直接引用某些原始字段、魔法值、散落常量。
- 是否限制跨层依赖、反向依赖或违规模块调用。

通过标准：

- 常见“结构性坏味道”能在静态阶段被拦住。

## 7. 契约与测试检查项

### 7.1 是否具备基础自动化验证

检查项：

- 关键模块是否有单元测试。
- 核心链路是否有集成测试。
- 是否有最小 smoke test。

通过标准：

- 关键变更至少会触发一层自动化验证。

### 7.2 是否具备契约测试

检查项：

- 前后端接口是否校验请求和响应契约。
- producer / consumer 是否校验事件契约。
- schema 默认值、字段定义、渲染能力、映射规则是否有 contract test。
- 三方适配层是否有输入输出契约测试。

通过标准：

- 边界协议不一致时，测试阶段能失败。

### 7.3 是否具备迁移验证

检查项：

- 数据库 migration 是否有回归验证。
- 历史数据兼容逻辑是否有 migration test。
- 旧版本事件、旧版 payload、旧配置结构是否有升级验证。

通过标准：

- 结构升级不是只“改定义”，而是验证历史数据能被正确消费。

### 7.4 是否具备三方系统验证能力

检查项：

- 是否有 sandbox 测试环境。
- 是否有 mock server 或回放测试。
- 是否覆盖成功、失败、重试、乱序、重复回调等边界行为。

通过标准：

- 三方协议变化或不稳定行为能在预发阶段尽早暴露。

## 8. CI 与发布门禁检查项

### 8.1 CI 是否具备阻断能力

检查项：

- 类型检查失败是否阻止合并。
- 契约测试失败是否阻止合并。
- 生成产物不一致是否阻止合并。
- 核心集成测试失败是否阻止合并。

通过标准：

- “本地看起来差不多”不能绕过流水线。

### 8.2 是否具备发布前门禁

检查项：

- 是否有部署前 smoke test。
- 是否有数据库变更前检查。
- 是否有三方系统连通性或沙箱校验。
- 是否有 feature flag、灰度、回滚预案。

通过标准：

- 高风险变更发布前有结构化确认，而不是人工口头确认。

## 9. 运行时保护检查项

### 9.1 是否具备外部不确定性防护

检查项：

- 是否有重试、超时、熔断机制。
- 是否有幂等设计。
- 是否有死信队列或失败补偿机制。
- 是否有关键链路监控和告警。

通过标准：

- 即使静态验证无法覆盖，运行时风险也能被观测、隔离和恢复。

### 9.2 是否具备变更后的观测能力

检查项：

- 是否能定位新版本引入的异常。
- 是否能关联接口、数据库、事件、三方请求日志。
- 是否能在异常放大前及时发现。

通过标准：

- 系统对异常具备足够透明度，不依赖用户投诉后才发现问题。

## 10. 分场景最低落地建议

### 10.1 纯前端项目

最低建议：

- 有 schema / spec 作为源头。
- UI、校验、默认值、渲染从源头派生。
- 有类型检查和 contract test。
- 有关键页面 smoke test。

### 10.2 前后端项目

最低建议：

- 有 API contract 作为统一源头。
- 有生成类型或 SDK。
- 有前后端契约测试。
- CI 阻止 breaking change 直接合并。

### 10.3 含数据库项目

最低建议：

- schema / migration 为唯一源头。
- 结构变更配套 migration test。
- 关键查询和写入路径有回归验证。

### 10.4 含三方对接项目

最低建议：

- 有 adapter layer 隔离三方协议。
- 有 sandbox / mock / replay 至少一种验证手段。
- 有重试、幂等、告警、回滚机制。

## 11. 项目成熟度判断

可以用以下三个层级判断项目当前成熟度：

### Level 1：文档约束阶段

特点：

- 有 rules，有说明文档。
- 主要依赖开发者和 AI 自觉遵守。
- 结构变更仍然容易漏改。

结论：

- 可作为起点，但不稳定。

### Level 2：结构约束阶段

特点：

- 已建立单一源头。
- 有类型检查、派生结构、基础契约测试。
- 大量漏改问题能在本地或 CI 暴露。

结论：

- 这是推荐的基本盘。

### Level 3：闭环约束阶段

特点：

- 从源头、派生、验证、发布到运行形成闭环。
- 对数据库、异步系统、三方系统都有专门门禁和保护。
- 不完整修改很难穿透到线上。

结论：

- 这是复杂系统和 AI 深度协作开发的理想状态。

## 12. 最终目标

这份清单的最终目标不是增加流程负担，而是建立一种稳定工程能力：

- 让修改尽量从源头进入。
- 让关联实现尽量自动派生。
- 让漏改尽量在更早阶段失败。
- 让系统在复杂迭代和 AI 协作中仍保持一致性。

如果一个项目能持续用这份清单补齐短板，它就会逐步从“依赖经验和提醒”转向“依赖结构和门禁”。
