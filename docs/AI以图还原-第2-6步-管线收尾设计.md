## 总编排（runtime 顺序 ≠ 步骤编号）

步骤编号是**设计顺序**；实际运行入口为 **`runRestoreAstFromDesignImage`**（HTTP 以图创建版式 · 方案 2）：

```
设计图
  │  第 4 步 · prompt → LLM
  ▼
RestoreAstDocument { theme, tree }            ← AI 输出（JSON）
  │  第 1 步 · astToTemplate（含 blockId↔astPath 映射；容器 title → blockMeta.name）
  ▼
template.json(占位 src) + tokenPresets.json
  │  第 2 步 · validate（结构绊线，按构造应永过）
  ▼
  │  第 3 步 · 资产回填（query → 搜索 + 验活 → 回填 src）
  ▼
  │  第 6 步 · 保真度门（启发式 / 视觉 diff）
  ▼
交付
  └─ 任一环失败 → 第 5 步 · 按节点定点修复（≤3 轮）→ 回到对应环 / 保底交付
```

> **方案 1（mjs patch）** 已从代码库移除；弹窗单选暂留供未来试验，当前仅 **RestoreAst** 可执行。

## 第 2 步 · 出口绊线（校验）

**做什么**：`astToTemplate` 产物在落盘前过一遍校验。**按构造应永过**；报错即 builder/派生的 bug → **改库，不修产物**（结构由程序保证，不靠 AI 兜）。

**建议形态**：

```ts
function assertValidByConstruction(template: EmailTemplate): {
  qualityIssues: ValidationIssue[];   // 非阻断，交第 6 步质量门
};
```

**行为**：

| 校验结果 | 处理 |
|---|---|
| blocking（结构非法：非法 type、缺必填、对象形态错…） | **抛 `AstAssemblerBug`**：记 node id + issue path 到日志,中断管线。这不该发生；发生即组装器有 bug,去补 builder/派生,**不重试、不丢给 AI** |
| 非 blocking（质量类） | 收进 `qualityIssues`,不中断,传给第 6 步 |

- **入参**：第 1 步产出的 `template`（内存对象）。
- **出参**：非阻断质量问题清单（正常多为空）。
- **复用**：`validateTemplateFromDisk`（`src/lib/templateTreeAdapter.ts:217`）/ `validateTemplate`（`src/lib/validate.ts:1453`）/ `blockingValidationIssues`（`src/lib/validate.ts:70`）——与编辑器同一套,不另写。
- **决策**：
  - 空串 src（资产占位）在结构层**合法**（placeholderSrc 属保真问题,不在此拦）→ 本步可在资产回填**之前**跑。
  - blocking 抛错而非进修复循环,是本架构的核心断言：**结构正确性由程序保证,不靠 AI 兜**。
