# 源头驱动的契约式开发

## 1. 定义

源头驱动的契约式开发，是一套面向长期迭代项目的通用工程方法论。

它的核心目标不是单纯提升开发效率，也不是依赖 prompt、skills、rules 去“提醒”开发者或 AI 少犯错，而是通过代码结构、类型系统、契约定义、自动派生、测试与发布门禁，尽可能把“不完整修改”拦截在更早阶段。

这套方法论适用于前端、后端、数据库、消息系统、三方系统对接，以及包含多模块协作的复杂业务系统。

一句话概括：

> 任何业务变更，优先修改源头定义；所有相关实现从源头派生；不完整修改必须在编译、校验、测试或发布门禁阶段失败，而不是在运行中潜伏。

## 2. 要解决的问题

在实际项目中，最危险的问题往往不是显性的语法错误，而是以下这类“漏改型错误”：

- 改了接口结构，但忘了同步前端调用方或后端实现。
- 改了数据库字段，但忘了同步查询、校验、DTO、序列化逻辑。
- 改了状态流转规则，但忘了同步页面按钮、审批逻辑或消息处理器。
- 改了三方系统的映射规则，但忘了同步适配层、回调处理和容错逻辑。
- 改了一个配置定义，但忘了同步渲染、表单、校验、导出或迁移逻辑。

这类问题的共同特征是：

- 代码可能能通过基础检查。
- 页面可能能打开，接口可能能返回。
- 主路径可能可用，但边缘路径和联动路径已经失真。
- 只有在特定条件下才会暴露，容易成为隐藏 bug。

传统依赖文档、经验、代码评审和人工记忆的方式，并不能稳定解决这个问题。AI 参与开发后，这类问题会被进一步放大，因为 AI 很擅长局部修改，但不天然可靠地掌握所有隐式依赖。

## 3. 核心主张

源头驱动的契约式开发，包含五条核心主张：

### 3.1 任何修改先改源头

任何需求变更都应先定位其“源头定义”。

例如：

- 改接口，先改接口契约。
- 改数据结构，先改 schema 或 migration。
- 改业务流程，先改状态机或领域规则。
- 改事件格式，先改 event contract。
- 改三方映射规则，先改 adapter contract。

不应优先修改中间表现层或消费层，再寄希望于其他相关模块自行跟上。

### 3.2 同一业务事实只能有一个真相来源

同一件业务事实，不应在多个模块中以平行方式维护多份定义。

例如：

- 字段结构不应在接口文档、前端表单、后端 DTO、数据库映射中各写一份。
- 状态枚举不应在前端、后端、任务系统中以散落常量方式重复维护。
- 三方回调字段不应在多个处理器中各自解析并各自理解。

如果同一件事有多份“真相”，那么每次迭代都必然存在同步遗漏的风险。

### 3.3 派生优于双写

凡是可以从源头自动生成、自动推导、统一派生的内容，就不应通过人工重复维护。

例如：

- 从 OpenAPI / Proto 生成 SDK、类型、mock、服务桩。
- 从数据库 schema 生成 ORM 类型、校验器、查询映射。
- 从表单或组件 schema 生成配置面板、默认值、校验逻辑。
- 从状态机定义派生按钮状态、允许操作、后端流转约束。

重复并不可怕，人工维护的重复才危险。

### 3.4 非法状态不可轻易表达

应尽量通过类型系统、封闭枚举、schema 校验、领域模型约束，让错误状态难以进入系统。

例如：

- 用联合类型或枚举表示状态，而不是随意字符串。
- 用严格 schema 校验外部输入，而不是直接信任 payload。
- 用明确的 Optional / Nullable 边界，而不是让空值到处传播。
- 用领域模型收敛业务规则，而不是让规则散落在控制器、页面、任务和消息消费者中。

目标不是“出错了再处理”，而是尽可能让错误状态在结构上不成立。

### 3.5 让不完整修改尽早失败

最佳状态是编译失败，其次是静态校验失败、契约测试失败、集成测试失败、发布门禁失败。

原则是：

- 能在生成期失败的，不留到编译期。
- 能在编译期失败的，不留到测试期。
- 能在测试期失败的，不留到线上。
- 无法静态验证的问题，必须在发布门禁和运行时保护中被监控和兜底。

## 4. 四层结构模型

为了让这套方法论可以跨项目复用，可以抽象为四层：

### 4.1 源头层

源头层负责定义系统中的“真相”。

典型内容包括：

- API contract
- GraphQL schema
- Proto definition
- database schema / migration
- domain model
- state machine
- event schema
- permission model
- third-party adapter contract

源头层的原则是：

- 关键事实必须集中定义。
- 任何业务变更先进入源头层。
- 其他模块不应各自维护一套等价定义。

### 4.2 派生层

派生层负责从源头自动得到其他实现。

典型内容包括：

- generated types
- generated client / sdk
- validators
- registry
- mappers
- form or panel metadata
- mock data contracts
- documentation artifacts

派生层的原则是：

- 优先使用生成和统一派生，减少手工同步。
- 所有派生产物都必须可重复生成。
- 生成结果应可在 CI 中校验是否过期。

### 4.3 验证层

验证层负责拦截“不完整修改”。

典型机制包括：

- type checking
- exhaustive checking
- lint rules
- unit tests
- contract tests
- integration tests
- migration tests
- sandbox tests

验证层的原则是：

- 越靠前的验证越优先。
- 验证要围绕“关联模块是否同步”来设计，而不仅是围绕“单模块是否可运行”。
- 契约测试和集成测试要覆盖跨模块链路。

### 4.4 运行层

运行层负责处理无法完全通过静态阶段覆盖的问题。

典型机制包括：

- idempotency
- retry policy
- timeout and circuit breaker
- dead letter queue
- feature flag
- observability
- alerting
- rollback plan

运行层的原则是：

- 外部不确定性必须显式隔离。
- 高风险操作必须可监控、可回滚、可降级。
- 三方对接、异步系统、网络故障不应只靠“测试通过”来假定安全。

## 5. 通用落地方式

这套方法论不是某个技术栈专属能力，而是一套可映射到不同系统的设计方式。

### 5.1 前端项目

适用场景：

- 低代码编辑器
- 配置驱动页面
- 表单系统
- 设计系统
- 权限驱动界面

建议方式：

- 用 schema 或 component spec 作为单一源头。
- 配置面板、默认值、校验、渲染、导出统一从源头派生。
- 所有组件能力通过 registry 注册，而不是散落条件分支。
- 对所有 spec 执行 contract test，验证默认值、字段定义、渲染能力的一致性。

### 5.2 前后端协作

适用场景：

- REST API
- GraphQL
- gRPC
- 微服务之间接口调用

建议方式：

- 用 OpenAPI、GraphQL schema 或 Proto 作为接口契约源头。
- 从契约生成前端 SDK、服务端类型、mock、测试数据。
- 对 breaking change 建立自动检测。
- 在 CI 中校验契约变更是否同步更新调用方和实现方。

### 5.3 数据库相关系统

适用场景：

- 业务系统
- 中后台系统
- 交易系统
- 报表系统

建议方式：

- 以 migration 或 schema 文件为唯一源头。
- 尽量从 schema 派生 ORM 类型、字段映射、校验逻辑。
- 对 schema 变更建立回归测试和 migration test。
- 禁止在多个层里散落维护字段含义和默认规则。

### 5.4 领域逻辑复杂的系统

适用场景：

- 审批流
- 订单系统
- 支付系统
- 工单系统
- 工作流系统

建议方式：

- 用状态机、聚合根、领域服务收敛业务规则。
- 页面、接口、任务、消息消费者统一走领域层。
- 对状态流转建立穷尽检查和契约测试。
- 禁止在多个入口点重复编写同一业务规则。

### 5.5 消息和事件驱动系统

适用场景：

- MQ
- 事件总线
- 订阅分发
- 异步任务编排

建议方式：

- 所有事件必须有明确 schema 和版本策略。
- producer 和 consumer 共享同一事件契约。
- 对兼容性变更建立 contract test。
- 对事件消费失败建立重试、幂等和死信策略。

### 5.6 三方系统对接

适用场景：

- 支付网关
- 电商平台
- ERP / CRM 对接
- OAuth / SSO
- webhook / callback 平台

建议方式：

- 建立独立 adapter 或 anti-corruption layer。
- 外部字段不直接渗透到内部领域模型。
- 用 adapter contract 描述映射关系、字段规则、错误码语义。
- 用 sandbox、mock server、response replay 做边界验证。
- 用告警、幂等、熔断、回滚计划处理真实外部不确定性。

## 6. “硬约束”应该如何理解

源头驱动的契约式开发追求“尽可能硬”，但不应把“硬约束”狭义理解为“只有编译失败才算硬”。

更准确的理解应是：

- 能通过生成期阻断的，属于第一层硬约束。
- 能通过编译期阻断的，属于第二层硬约束。
- 能通过静态校验和测试阻断的，属于第三层硬约束。
- 能通过发布门禁阻断的，属于第四层硬约束。
- 能通过运行时保护收口风险的，属于最后一层防线。

理想目标不是宣称“所有问题都要在编译期解决”，而是：

> 建立一套从源头定义到发布运行的闭环体系，让漏改越来越难穿透到线上。

## 7. 与 skills / rules 的关系

skills、rules、prompt 仍然有价值，但它们的角色应该是辅助层，而不是主要保障层。

建议理解为：

- skills / rules 负责告诉 AI“应该如何工作”。
- 源头驱动的契约式开发负责规定“系统允许怎样被修改”。
- 类型系统、契约、测试、CI 负责保证“错误修改不能轻易通过”。

因此，推荐的优先级是：

1. 先建立单一源头和派生结构。
2. 再建立类型、契约测试、集成测试和发布门禁。
3. 最后用 skills / rules 去提醒 AI 按正确路径完成修改。

如果一个项目只依赖 rules，而没有结构层面的约束，那么无论是人还是 AI，都仍然可能在局部修改中遗漏全链路影响。

## 8. 面向 AI 协作开发的价值

这套方法论在 AI 开发场景中尤其重要，因为 AI 的典型行为模式通常是：

- 优先处理眼前文件。
- 更擅长局部补丁，而不是主动维护隐式依赖。
- 在没有硬性失败信号时，容易默认“当前修改已经足够”。

因此，AI 时代最关键的不是不断增加提示词长度，而是把系统改造成：

- 修改必须经过源头定义。
- 相关实现从源头派生。
- 漏改会在自动化流程中失败。

这样一来，AI 的任务就从“靠理解和记忆维护一致性”，转变为“在明确结构和门禁中完成修复”。

## 9. 团队执行原则

为了让这套方法论真正落地，建议团队形成统一执行规则：

### 9.1 变更入口原则

- 任何需求变更，先识别源头定义。
- 未修改源头定义，不应直接改下游消费层。
- 源头变更后，必须检查所有派生产物和关联模块。

### 9.2 禁止双写原则

- 任何字段结构、状态规则、事件格式、权限规则，不应散落定义多份真相。
- 如发现双写，应优先收敛结构，而不是依赖文档提醒同步。

### 9.3 契约先行原则

- 所有模块边界必须显式化。
- 接口、事件、消息、三方适配都应有可执行契约。

### 9.4 自动验证原则

- 关键链路必须有自动化验证。
- 验证不仅覆盖模块内逻辑，还要覆盖模块间一致性。

### 9.5 发布门禁原则

- 高风险变更必须通过预发布校验。
- 涉及数据库、消息系统、三方系统的改动，必须具备可观测性和回滚方案。

## 10. 通用检查清单

当发生任何较大变更时，可以使用以下检查清单：

### 10.1 源头检查

- 这次变更的真正源头定义是什么？
- 是否先修改了源头，而不是先改消费层？
- 是否还有其他地方在平行维护同一事实？

### 10.2 派生检查

- 哪些类型、SDK、表单、映射器、文档、mock 应从源头重新生成？
- 当前是否存在人工同步维护的重复结构？
- 是否可以进一步收敛为自动派生？

### 10.3 影响面检查

- 哪些前端页面、后端模块、数据库查询、消息消费者、三方适配器会受影响？
- 是否存在低频但高风险的边缘路径？
- 是否存在定时任务、补偿任务、回调处理器等非主链路消费者？

### 10.4 验证检查

- 是否有类型检查能发现遗漏？
- 是否有契约测试能发现边界不一致？
- 是否有集成测试能覆盖关键跨模块链路？
- 是否有 sandbox / replay / migration test 覆盖外部依赖和数据变更？

### 10.5 发布检查

- 是否具备回滚策略？
- 是否具备必要的监控与告警？
- 是否具备 feature flag 或灰度发布能力？

## 11. 适合作为团队共识的五句话

以下五句话可以直接作为团队工程原则：

1. 任何变更先改源头定义，不先改消费层。
2. 同一业务事实只能有一个真相来源。
3. 所有关联实现尽量从源头派生，而不是人工同步。
4. 能在编译期失败的，绝不留到运行期。
5. 不能在编译期失败的，必须在契约、集成、发布门禁中失败。

## 12. 结语

源头驱动的契约式开发，不是某个框架技巧，也不是某种只适用于前端的模式，而是一套面向复杂系统和长期迭代的工程约束思想。

它的意义在于：

- 把“靠人记住所有关联修改”转变为“靠系统结构约束修改路径”。
- 把“修改完成后再找 bug”转变为“修改不完整时尽早失败”。
- 把“AI 可能漏改”转变为“漏改后无法轻易蒙混通过”。

对于 AI 参与越来越深的开发环境，这种方法论会从优秀实践，逐步变成基础能力。
