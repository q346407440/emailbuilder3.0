# 可拔插式代码开发规范

## 1. 定义

可拔插式代码开发，是一套面向长期迭代项目的通用工程方法论。

它的核心目标不是先做「大模块拆分」或「重型插件框架」，而是在**写每一个功能、每一个函数**时，默认让代码具备这样的性质：

- **可加**：新增一段逻辑，Prefer 新增函数并在调用处装配，而不是往旧函数里塞分支。
- **可换**：同职责存在多种实现时，Prefer 参数、策略或查表切换，而不是复制粘贴改多处。
- **可删**：去掉某个功能或某段逻辑后，残留引用应能被编译、静态检查或测试尽早发现。

这套方法论适用于前端、后端、数据库、消息系统、三方系统对接，以及包含多模块协作的复杂业务系统。粒度上，它既适用于单个纯函数，也适用于由多个函数组合而成的一个功能点；**不要求**每次改动都先做大架构设计。

一句话概括：

> 每个函数只做一件事、只依赖显式输入；功能由函数组合而成；增删改逻辑时，尽量动局部单元，而不是不断拉长旧函数。

本规范与《源头驱动的契约式开发》《变更先改源头执行规范》《AI Coding 硬约束落地清单》配套使用：

- **源头驱动**解决“改一处时，全链路如何保持一致”。
- **可拔插**解决“加一段、删一段逻辑时，旧代码如何不被越改越缠”。

二者并不冲突：函数的类型签名、输入输出 schema、策略表条目，本身就是最小粒度的“契约”。

## 2. 要解决的问题

在实际开发和 AI 协作中，系统逐渐变得难以维护，往往不是因为缺少“大模块”，而是因为**函数层面的耦合**：

- 一个函数里同时做校验、转换、持久化、发通知，改其中一步要理解整段逻辑。
- 新增一种业务分支，不是新增一个小函数，而是在 5 个已有函数里各加一个 `if`。
- 两个功能共用一段逻辑，但复制粘贴了两份；修 bug 只改了一处。
- 删除实验功能时，主函数里还留着半段 dead branch，或 helper 仍被间接引用。
- 为了“先跑起来”，把本可独立的 20 行逻辑写进 300 行的入口函数，之后谁都不敢动。

这类问题的共同特征是：

- 单个文件、单个函数看起来“能工作”。
- 每加一个功能，旧函数持续变长，而不是系统多出几个清晰的小单元。
- 删一段逻辑时，不知道还有没有隐式依赖。
- 测试只能测整条链路，无法单独验证某段规则。

传统依赖“这段逻辑先放这里，以后再抽”的习惯，会把成本推迟到以后。AI 参与开发后，这个问题更明显：AI 很擅长在现有函数末尾追加代码，但不天然会主动拆函数、减耦合。

可拔插式开发的目标，是把“改一个功能要动很多旧函数”变成“新增一个函数、替换一个函数、删除一个函数，影响面可控”。

## 3. 核心主张

可拔插式代码开发，包含六条核心主张。它们应体现在**每一次写函数、每一次加功能**时，而不是只在架构评审时才提起。

### 3.1 一个函数一件事

每个函数只承担**一个清晰职责**，并能用一句话说明它在做什么。

好的信号：

- 函数名能准确描述单一动作：`parsePayload`、`buildOrderSummary`、`validateEmailFormat`。
- 函数长度通常可控；若变长，是因为数据步骤多，而不是职责变多。
- 读函数时，不需要同时理解无关业务分支。

坏的信号：

- 函数名含 “And”“HandleAll”“ProcessEverything” 等模糊语义。
- 一个函数里穿插“顺便”做的校验、日志、缓存、通知。
- 为了复用，把两件不相关的事塞进同一个 helper。

原则：

> 不是“文件要小”，而是“函数不要兼任多个角色”。

### 3.2 输入输出显式，拒绝隐式依赖

函数应通过**参数和返回值**与外界协作，而不是通过隐式渠道读写字段。

显式依赖包括：

- 函数参数
- 返回值 / Result 类型
- 注入的 callback、strategy、repository（作为参数传入）

隐式依赖包括：

- 读写全局变量、模块级 mutable 状态
- 直接 import 并调用与当前职责无关的具体实现
- 依赖调用顺序、环境变量、当前时间等未出现在签名中的前提

原则：

- 看到函数签名，应能大致知道它需要什么、产出什么。
- 替换实现时，Prefer 换传入的依赖或换被调用的那个函数，而不是改调用方内部细节。

### 3.3 纯逻辑与副作用分离

即使不立刻拆文件，也应在**函数级别**区分：

- **纯逻辑函数**：给定输入，返回输出；不读写外部世界。
- **副作用函数**：发请求、写库、写文件、改 UI、发消息。

建议方式：

- 业务规则、格式化、聚合、校验 → 纯函数。
- IO 与框架调用 → 薄 wrapper，内部调用纯函数处理结果。

收益：

- 纯函数可单独单测，不需要 mock 整个环境。
- 换存储、换 UI 框架时，大量规则函数可原样保留。
- 删除某个 IO 路径时，不会误伤规则本身。

### 3.4 组合优于在一个函数里加分支

新增能力时，优先**新增函数并在调用处组合**，而不是在旧函数里不断追加 `if / else / switch`。

推荐模式：

```text
// 不推荐：旧函数越来越长
function renderItem(item) {
  if (item.type === 'a') { ... }
  else if (item.type === 'b') { ... }
  else if (item.type === 'c') { ... }  // 每次新增类型都改这里
}

// 推荐：按类型拆函数，调用处查表
const renderers = { a: renderA, b: renderB, c: renderC };
function renderItem(item) {
  return renderers[item.type](item);
}
```

原则：

- 主调度函数保持短小：查表、循环、组合、传递上下文。
- 具体规则下沉到独立函数；新增类型 = 新增函数 + 注册到表，而不是修改旧函数语义。
- 同一 `switch` 在多个文件重复出现时，说明应提取共享的策略表或共享纯函数。

### 3.5 功能是可拔插函数的组合

“功能”不是一个大函数，而是**若干小函数 + 一条装配路径**。

写一个功能时，默认问自己：

1. 哪几步是纯逻辑？能否各自成函数？
2. 哪几步是副作用？能否压到边界？
3. 如果明天删掉这个功能，能否只删一组函数和一处装配？
4. 如果要在测试环境禁用这个功能，能否不调用那组函数即可？

禁止习惯：

- 为一个小需求，给已有核心函数新增特殊分支，导致所有调用方承担新语义。
- 新功能依赖旧函数的副作用顺序，但没有文档、没有类型、没有测试约束。

### 3.6 删除必须可检测

删除函数、删除分支、删除功能时，系统应能**尽早告诉你还有没有残留**。

优先依赖：

- 类型与 export：删掉 export 后，残留 import 编译失败。
- unused code 检测：lint、dead code analysis。
- 单测：删掉函数后，对应测试文件应删除或失败，而不是留空壳。
- 策略表 / registry：移除条目后，穷尽检查或契约测试失败。

禁止依赖：

- “这个 helper 大概没人用了吧”
- “先把调用注释掉，函数先留着”
- “分支走不到就行，代码不用删干净”

## 4. 三个粒度：函数、功能、装配

可拔插不是只有“架构层”才需要考虑。日常编码可在三个粒度上理解：

### 4.1 函数（最小单元）

最小可拔插单元。要求：

- 单一职责
- 显式输入输出
- 可单独测试
- 删改时不牵连无关函数

这是本规范的**默认落脚点**：先保证函数可拔插，再谈更大结构。

### 4.2 功能（一组协作函数）

一个用户可感知或一个业务步骤对应的能力，通常由多个函数完成：

- 校验函数 + 转换函数 + 持久化 wrapper
- 解析函数 + 规则函数 + 渲染 wrapper

要求：

- 功能内部通过组合实现，而不是一个函数包到底。
- 功能之间 Prefer 松散耦合：A 功能删除，不应迫使重写 B 功能的内部逻辑。

### 4.3 装配（调用入口）

页面事件、controller、route handler、CLI 命令、定时任务入口等。

要求：

- 装配层只做**调用顺序与上下文传递**，不写大段业务规则。
- 装配层变长时，首先怀疑：是否应把逻辑下沉为函数。
- 新增功能时，装配层 Prefer 多一行调用，而不是把旧入口函数改写成更长版本。

当多个功能共享同一装配入口时，才需要上升到 registry、插件表、策略表等结构——那是**函数数量变多之后的自然结果**，不是前提。

## 5. 写函数时的默认习惯

以下习惯应在日常编码中默认执行，而不是等 refactor 阶段再补。

### 5.1 动手前先问四个问题

1. 这个函数**只做一件事**吗？
2. 它的输入输出是否**显式**？
3. 它能否在**不启动全系统**的情况下单独测试？
4. 如果**删除**它，影响范围是否清晰？

四个问题有任一答不清楚，先不要继续堆代码，先拆函数或改签名。

### 5.2 出现分支时，先想“要不要新函数”

当准备在现有函数里加 `if` 时，默认先评估：

- 这个分支是不是一种**独立规则**？是 → 新函数。
- 这个分支会不会在别处也需要？是 → 共享纯函数或策略表。
- 这个分支是不是临时实验？是 → 独立函数 + 独立装配，便于整段删除。

只有“一行以内、且不会扩散”的分支，才可以留在原函数中。

### 5.3 复制粘贴是信号，不是解法

两处出现相似代码时：

- 第一次：可以写两个独立函数，即使暂时相似。
- 第二次：提取**稳定的公共纯函数**，而不是把两处都改得更复杂。
- 禁止：为了少写一个函数，让 A 函数偷偷适配 B 场景的特殊参数。

目标不是“绝对不重代码”，而是“重代码处应是稳定、可测试、可替换的小函数”。

### 5.4 副作用函数要薄

副作用函数的理想形态：

```text
read input → call pure function → write result / trigger side effect
```

如果副作用函数里出现大量业务规则，应把规则抽到纯函数，副作用函数只负责取数和写回。

### 5.5 命名即边界

函数名应让人判断能否安全调用、替换、删除：

- `computeX`、`validateX`、`formatX`：通常是纯逻辑，可拔插性高。
- `handleX`、`processX`：若含义过宽，往往已经混入多职责，应继续拆。

## 6. 写功能时的默认习惯

### 6.1 新增功能：Prefer 新增，而不是改旧

默认顺序：

1. 写清该功能需要的纯逻辑函数。
2. 写必要的 IO wrapper。
3. 在装配入口增加调用。
4. 补该功能对应的单测。

禁止顺序：

- 先在核心入口函数里加分支，让旧功能承担新语义。
- 先改共享 helper 的返回值结构，迫使所有旧调用方一起改。

### 6.2 修改功能：Prefer 换函数，而不是改语义

当规则变化时：

- 若只影响一种场景：Prefer 新函数 + 切换调用。
- 若影响所有场景：修改纯函数，但保持签名稳定；必要时用新函数名并迁移调用方。

避免：

- 在旧函数里加参数 `mode: 'old' | 'new'`，让一份实现长期承担两套语义。

### 6.3 删除功能：整段拔，不要留半套

删除顺序：

1. 从装配入口移除调用。
2. 删除该功能专属函数与测试。
3. 从策略表 / registry 移除条目（若有）。
4. 运行编译、lint、测试，确认无残留 import 或 dead branch。

完成标准不是“界面上看不到了”，而是“代码里不存在对该功能的可执行路径”。

## 7. 与源头驱动方法论的关系

| 维度 | 源头驱动 | 可拔插 |
|------|----------|--------|
| 主要问题 | 改一处漏改多处 | 加一段逻辑越改越缠 |
| 粒度 | 契约、schema、真源定义 | 函数、功能、装配 |
| 核心手段 | 单一真源、派生、尽早失败 | 单一职责、显式 IO、组合 |
| 典型产物 | contract、generated types | pure functions、strategy map、thin wrapper |

推荐关系：

1. **契约落到函数签名**：源头定义的字段、枚举、规则，最终应体现在纯函数的输入输出类型上，而不是散落在多个大函数内部。
2. **派生降低重复**：能从 schema 生成的校验函数、映射函数，不要手写多份等价逻辑。
3. **验证从单测开始**：一个函数一份契约，比事后靠集成测试才发现“删错了 helper”更早、更便宜。

简言之：

> 源头驱动保证“改的是真源”；可拔插保证“动的是一小段，不是整张网”。

## 8. 分场景下的函数级实践

### 8.1 前端与 UI

- 事件 handler 保持薄：只取 DOM 值、调纯函数、把结果交给 state / API wrapper。
- 格式化、校验、可见性判断 → 纯函数。
- 复杂组件内部：render 逻辑、规则逻辑、数据转换逻辑分成不同函数，而不是一个超大组件函数。

### 8.2 后端接口与领域逻辑

- controller 只做参数解析、调用领域函数、返回响应。
- 同一业务规则不要在 controller、service、consumer 各写一份；提取共享纯函数。
- repository 只做读写，不做业务判断。

### 8.3 校验与映射

- 每个字段规则、每种映射各自成函数，或由 schema 派生。
- 新增字段：新增函数或新增派生规则，而不是修改一个 `validateAll` 巨函数。

### 8.4 消息与异步任务

- 每种事件处理拆成独立 handler 函数。
- handler 内：`parse → applyRule (pure) → persist (side effect)`。
- 关闭某类处理：删 handler 与注册项，而不是在主 consumer 里留空分支。

### 8.5 脚本与迁移

- 每个脚本只做一种变更；步骤函数化，便于单测和复用。
- 脚本复用领域纯函数，不复制键表或规则字符串。

## 9. AI Coding 场景专用要求

### 9.1 不允许用“继续追加”代替设计

AI 不应默认在现有函数末尾追加几十行代码完成任务。

应先判断：

- 这段逻辑是否应成为新函数？
- 是否应提取纯函数？
- 是否只是在重复已有函数的部分行为？

### 9.2 不允许制造“半删除”

AI 删除功能时，不得只删 UI 或只删入口，留下：

- 未被调用的 helper
- 永远为 false 的分支
- 只服务已删功能的测试夹具

### 9.3 不允许共享 helper 偷偷改变语义

AI 不应为了修 A 场景，直接修改被 B、C 共用的函数行为，而不评估所有调用方。

Prefer：

- 新函数
- 新参数（有默认值且语义清晰）
- 显式迁移所有调用方

### 9.4 要求能说明“拔插点”

对于新增或修改，AI 应能说明：

- 新增了哪些函数？各自职责是什么？
- 删功能时会删哪几个函数、改哪一处装配？
- 哪些是纯函数，哪些有副作用？

如果无法说明，说明代码尚未按可拔插方式组织。

## 10. 通用检查清单

### 10.1 函数级

- 能否用一句话说清该函数职责？
- 签名是否表达了全部必要输入？
- 是否存在隐式全局依赖？
- 能否在不 mock 整个系统的情况下单测核心逻辑？
- 删除该函数时，编译器/lint 能否暴露残留引用？

### 10.2 功能级

- 该功能是否由多个小函数组合，而不是一个巨函数？
- 新增该功能是否主要“新增代码”，而不是“改很多旧函数”？
- 关闭该功能是否只需移除一组调用？

### 10.3 装配级

- 入口函数是否 mainly 做调度？
- 是否出现“入口函数越来越长”的趋势？
- 重复 `switch` 是否应收敛为策略表？

### 10.4 删除与替换

- 删逻辑后是否仍残留 dead code？
- 替换实现时是否只换一个函数或一张表？
- 测试是否同步删改？

## 11. 完成标准

一次编码任务只有满足以下条件，才可视为符合本规范：

- 新增逻辑 Prefer 体现为新函数或新装配，而不是无节制拉长旧函数。
- 业务规则 Prefer 落在纯函数或薄 wrapper 中，副作用边界清晰。
- 每个关键函数可单独理解和测试。
- 删除对应功能时，影响范围可列举，且无已知隐式依赖。
- 必要单测与静态检查已通过。

以下情况不能视为完成：

- 功能可用，但逻辑堆在入口大函数里。
- 删除功能后仍残留 helper、分支或测试空壳。
- 修一个场景导致多个无关场景行为变化。
- 只有端到端手工点测，没有任何函数级验证。

## 12. 评审要求

代码评审时，默认检查以下问题：

- 这次改动是在**新增/替换小函数**，还是在**继续拉长旧函数**？
- 是否存在本可拆出 pure function 却写在 IO 函数里的逻辑？
- 是否存在 duplicated `switch` 或 duplicated 规则？
- 若明天删除这段功能，能否干净拔掉？
- 函数名与签名是否 honest 地反映了职责？

如果这些问题回答不清楚，即使 demo 能跑，也不应直接通过评审。

## 13. 适合作为团队共识的五句话

1. 一个函数一件事，输入输出说清。
2. 新逻辑 Prefer 新函数，不在旧函数里无限加分支。
3. 规则写纯函数，IO 写薄 wrapper。
4. 功能是可拔插函数的组合，不是巨函数。
5. 删代码要删干净，靠编译和测试暴露残留。

## 14. 结语

可拔插式代码开发，不是先做大型插件架构，而是一种**编码时的默认姿势**：

- 写函数时，就让它可单独理解、测试、替换、删除。
- 写功能时，就用组合而不是堆叠。
- 写系统时，大结构自然从大量可拔插小单元中生长出来。

它与源头驱动的契约式开发结合后：

- **源头驱动**保证改的是真源，不会漏改关联面。
- **可拔插**保证每次改动动的是局部单元，不会把旧代码越缠越死。

对于 AI 参与越来越深的开发环境，函数级的可拔插习惯，比事后做大重构更便宜，也更可靠。
