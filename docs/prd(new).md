# 邮件模板编辑器第一期 prd

## 目标

接入 loyalty 内部运营平台，支持 loyalty 发送给商家的邮件使用新的邮件编辑器的模板

## 编辑器功能结构

编辑器功能结构分为两部分：

- 部分 1：模板列表
- 部分 2：编辑器页面

## 详细功能说明

### 邮件模板列表

#### 模板与版式的概念

模板列表的构成为模板与版式构成

结构为，1 个模板下，可存在多个版式的模板，且由于这多个版式共同属于一个模板，所以他们使用的是统一的数据变量

可以简单理解为，相同场景的模板，比如说新品推荐的模板。可能都是使用同一组商品列表。但是模板的排版布局是可以不一样。所以我这里称之为 1 个模板下有多个版式的概念

![邮件模板目录页：左栏模板列表 + 右栏版式详情](./prd-assets/project-prd-template-catalog-page.png)

#### 左侧模板列表

左侧模板列表。支持弹窗新建，新建后则创建一个有一个默认版式的模板记录

![左侧模板列表：搜索、发布筛选与模板卡片](./prd-assets/project-prd-catalog-sidebar.png)

![新建邮件模板弹窗](./prd-assets/project-prd-catalog-modal-create-template.png)

- 模板名称允许重复，名称限制 50 个字符
- 模板列表按创建时间倒叙排序
- 且模板列表支持模板名称模糊搜索，以及发布状态的快捷筛选

发布状态用于 loyalty 创建商家邮件发送活动时，允许选择，以及实际发送时允许通过的逻辑校验，未发布的模板不允许在商家邮件活动时选择和实际后端发送

#### 右侧：选中模板的版式列表

在右侧的选中的模板的版式列表处

**上部分：基础信息展示**

上部分是展示当前模板的一些基础信息，分别为：

- 模板标题
- 模板描述
- 模板的发信标题
- 模板的发信摘要
- 和变量数量

![右侧上部分：模板标题、说明、发信信息与变量数量摘要](./prd-assets/project-prd-catalog-detail-summary.png)

**模板层级操作**

且模板层级的操作支持：

- **模板信息编辑（弹窗）**
  - 分别编辑模板名称（50 个字符）必填
  - 模板的说明（100 个字符）非必填
  - 模板的发信标题（50 个字符）非必填
  - 模板的发信摘要（100 个字符）非必填
  - 模板的发信标题和发信摘要是对模板下的所有版式生效的。目前近对发送测试邮件生效。不影响实际的商家邮件活动时的邮件发信标题和摘要的逻辑

![编辑模板信息弹窗](./prd-assets/project-prd-catalog-meta-editor-modal.png)

- **发布和撤回发布**（做一个状态变更）
- **复制模板**
  - 将模板的所有信息，都复制一遍，文件级的复制，包括模板下的所有版式都复制

![复制邮件模板弹窗](./prd-assets/project-prd-catalog-modal-copy-template.png)

- **删除模板**（逻辑删除，弹窗二次确认）

![删除邮件模板二次确认](./prd-assets/project-prd-catalog-confirm-delete-template.png)

**版式列表**

版式列表，同样支持版式名称模糊搜索，以及仅查看已发布的

已发布的状态与模板一致，仅已发布的模板支持在创建 loyalty 的商家邮件活动时被选择，以及实际后端发送时允许通过发送

![版式列表：搜索、发布筛选与版式表格](./prd-assets/project-prd-catalog-design-list.png)

新增版式支持两种方案：

- 一种是空白方案，则创建一个空白的版式模板
- 一种是 AI 以图创建版式，详见下文「AI 以图创建版式」

![新版式弹窗（空白 / 以图创建）](./prd-assets/project-prd-catalog-modal-create-layout.png)

![新版式弹窗 · 以图创建（设计图上传 + 开始生成）](./prd-assets/project-prd-layout-create-modal-ai.png)

- 版式的名称必填，（50 个字符）
- 版式列表的排序同样也是创建时间倒叙

版式支持操作为：

- 编辑 or 点击版式名称，进入编辑器页，选中该版式
- 重命名则弹窗修改版式名称

![重命名版式弹窗](./prd-assets/project-prd-catalog-modal-rename-layout.png)

- 复制版式
  - 复制版式，相当于复制整个版式的内容。包括数据变量，和主题的绑定关系

- 发布 和撤回发布，变更发布状态
- 删除（逻辑删除，弹窗二次确认）

![删除版式二次确认](./prd-assets/project-prd-catalog-confirm-delete-layout.png)

##### 右侧配置面板 · 公共配置

每一个组件在右侧的组件配置面板中都有以下的几个配置：

- 内容、样式、布局、数据组、显隐

其中样式和布局，存在着 8 种组件都有的公共的配置

也就是：

- 外层容器 · 样式
- 与
- 外层容器 · 布局

![右侧配置面板「样式」页签 · 外层容器 · 样式（示例：选中文本区块）](./prd-assets/project-prd-config-panel-outer-wrapper-style.png)

![右侧配置面板「布局」页签 · 外层容器 · 布局（同上示例区块）](./prd-assets/project-prd-config-panel-outer-wrapper-layout.png)

这是因为这 8 种组件的基本构成，都是由组件自己本身，加上一个外层容器构成的

而这个外层容器支持以下的配置

**样式配置有**

- 容器背景色
- 圆角模式（四个统一还是分别配置）
- 圆角 px
- 描边模式（四边统一还是分别配置）
- 描边宽度
- 描边样式（实线虚线点线）
- 描边颜色

**布局配置有**

- 水平对齐（左中右）
- 垂直对齐（上中下）

用于控制外层容器控制中间的内容，在它的容器内的摆放外置的

一般情况下可能看不出区别。但是一旦通过自定义这个外层容器的宽高。使得里边的内容与容器不是贴边的。那么就可以看出效果了

- 宽度模式（跟随内容、铺满父级、自定义）
- 高度模式（跟随内容、铺满父级、自定义）

这里要注意，如果这个 block 的父级容器，比如 布局组件，或者栅格组件

的生效逻辑是铺满父级或跟随内容时

这个作为子级的 block 会存在一些宽高的配置限制

- 当父级的宽为跟随内容时，子级的宽不能选铺满父级。因为会造成死循环

同理

- 当父级的高为跟随内容时，子级的高不能选铺满父级，也是因为会造成死循环

- 容器内边距模式（四边统一，分别配置）
- 容器内边距数值

容器内边距其实就是 padding

讲完了block 都有的公共配置，那么就来到了每个 block 都有的特有的配置了

**栅格组件**

- 内容：
  - 支持启用背景图
  - 那么栅格的组件的外层容器会有覆盖一层背景图
  - 背景图地址
  - 背景链接地址（用于点击跳转的链接）
  - 背景图填充策略（裁切铺满、完整显示）
  - 背景画面位置（仅裁切铺满时支持配置）
  - 九宫格方向，控制背景图的裁切展示方向
  - 因为图片以短边裁切铺满视窗后，虽然默认是正中心展示。但是通过这个配置，可以让用户选择裁切展示的方向是九宫格的哪个方向
- 布局
  - 每行列数
  - 控制栅格一行有多少个格子
  - 间距
  - 控制栅格格子之间的上下和左右的gap
  - 单元格宽度模式（自动均分、固定宽度）
  - 自动均分指的是根据父级预留的宽度，实现均分宽度
  - 单元格高度模式（按行内容最大高度、固定高度）
  - 按行内容最大高度，指的是取单元格中，最大的那个单元格的高度。赋值给其他单元格
  - 栅格的每个单元格实际上宽高都必须是统一的
  - 所以此处配置的宽高都是给每个格子生效的
  - 且栅格的每个单元格实际上是一个垂直方向布局的单元格

### AI 以图创建版式

**定位**

AI 以图创建版式，是新建版式时的一种可选提效方式

生成的是版式结构与版式默认的主题样式初稿，不会自动生成业务变量的取值

运营仍需要对照设计图人工验收，并在编辑器里继续调整后，再保存区块、保存主题样式，再考虑发布

**入口**

与空白新建版式相同，有两个入口：

- 模板列表页，选中模板后的版式列表处，新建版式
- 编辑器顶栏，版式选择器处的新建版式

在「新版式」弹窗里，创建方式选择「以图创建」，带 AI 标记

![新版式弹窗 · 创建方式（空白方案 / 以图创建 AI）](./prd-assets/project-prd-catalog-modal-create-layout.png)

**填写与上传**

- 版式名称必填，（50 个字符）
- 设计图支持 JPG、PNG、WebP，单张不超过 10MB
- 上传设计图后，点击「开始生成」

![新版式弹窗 · 以图创建（设计图上传 + 开始生成）](./prd-assets/project-prd-layout-create-modal-ai.png)

**生成过程**

提交后，弹窗内展示分步进度，整段过程大约需要 2–8 分钟

生成过程中弹窗不可关闭，需等待完成

分步进度包括：

**识别视觉规格**

AI 看上传的设计图，先整体理解这封邮件长什么样，输出一份「视觉规格」JSON

主要包括：从上到下有哪些模块区域、主色副色和背景色、间距和字号档位、哪些地方需要摄影图或图标、各区域一句 layout 摘要等

这一步是在 **划区域、定规格**，还不是编辑器里最终的 block 树

**参考提示词**

以下为 **参考节选**，占位符（如 `{emailKey}`）由程序在运行时替换；**不要求实现逐字一致**。

```text
你是邮件模板视觉规格分析助手。只看设计图，只输出 JSON，不写模板代码。

须包含：description、colors、spacing、typography、emailRootBackground、
imageSlots、iconSlots、dividers、visualChecks、sections。

imageSlots 只写搜索条件（slotId、query、targetWidth、height），不写 URL。
iconSlots 写 pack（tabler / simple-icons / lucide）、iconQuery、colorHex 等。
sections 自上而下粗分区（sectionId 从 s1 递增），每区一句 summary，不要复制长正文。
只输出 JSON，不要 markdown 或解释。

运行参数：emailKey={emailKey}，displayName={displayName}，idPrefix={idPrefix}
（附带设计图）
```

**执行脚本并校验**

**检查视觉质量**

对照第一步识别出的视觉规格，检查生成结果有没有 **明显跑偏**

例如：图标是否该带外框、页脚字号是否过小、关键区域高度是否差太多等

有 **硬伤** 会触发重试；若最后一轮 **只剩建议项、没有硬伤**，则 **正常完成** 进入编辑器（不算保底交付，但进入编辑器后仍可能看到建议提示）

**参考提示词**

**落盘新版式**

校验或执行失败时，会自动重试，最多 3 次

**生成结果**

- 成功：关闭弹窗，自动进入新生成的版式编辑器，可在画布中看到还原结果；新版式默认未发布
- 失败：不落盘半成品，弹窗内提示失败，可换图或稍后重试

生成完成后，仍可能存在待确认项，须对照设计图继续改，不要直接发布

**校验与模板检查**

生成链路在「执行并校验」「检查视觉质量」「写入新版式」几步都会做检查，主要包括：

- 生成的版式结构是否符合编辑器规则（组件类型、允许填写的字段、嵌套关系等）
- 主题样式档位是否在约定范围内
- 对照第一步识别出的视觉规格，有没有明显跑偏（间距、字号、资产是否齐全等）

校验结果分几种情况：

- **全部通过**：结构与视觉检查均无 **须修复** 问题；写入新版式，进入编辑器
- **可自动修复或可重试**：前几轮仍有问题；系统自动修正，或带着问题重新生成，**继续下一轮**（最多 3 轮）。此阶段 **不会** 进入编辑器
- **仍有问题但已有可用产物（保底交付）**：见下节
- **没有可用产物**：创建失败，弹窗提示失败，**不落盘半成品**

**保底交付：什么情况下允许通过**

重试次数用尽（或系统判断继续重试也不会更好）时：**只要版式结构可用**——能在编辑器里打开、在画布上预览——就允许带着未解决问题进入编辑器，并提示运营建议人工确认后再发布。遗留的是规则、视觉还是素材问题，不改变这一判定。

**不允许通过（直接失败）—— 没有可交付的版式**

下列情况 **不允许进入编辑器**，弹窗提示失败，**不写新版式**：

1. **第一步识别视觉规格失败**：读不懂设计图，无法形成可用的视觉规格，后续无法继续
2. **结构不符合要求**：重试用尽后仍 **无法产出可预览的版式**（结构不合法、无法加载预览等）

**进入编辑器后的模板检查**

编辑器会做两个方面的模板检查：

1. 结构是否符合编辑器的要求
2. 配置的值是否符合编辑器的要求，或者是否有空值

**结构不符合编辑器的要求**

如果是结构不符合编辑器的要求，那么编辑器无法渲染当前的模板，需要用户切换其他模板

**配置值不符合编辑器的要求**
