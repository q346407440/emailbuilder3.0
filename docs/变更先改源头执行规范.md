# 变更先改源头执行规范

## 1. 目的

本规范用于约束团队成员和 AI 在处理需求变更、缺陷修复、结构调整时的执行顺序。

目标不是要求“改得更仔细”，而是要求：

- 所有变更优先进入源头定义。
- 所有关联修改按照固定路径展开。
- 所有关键验证在提交前完成。

本规范是《源头驱动的契约式开发》的操作层文档，可与《AI Coding 硬约束落地清单》配套使用。

## 2. 核心原则

执行本规范时，始终遵循以下五条原则：

1. 任何变更先定位源头定义。
2. 未修改源头前，不直接修改下游消费层。
3. 所有关联模块必须显式检查，不依赖记忆补漏。
4. 验证必须覆盖契约一致性，而不仅是局部可运行。
5. 无法完成全链路闭环时，不应视为变更完成。

## 3. 适用范围

本规范适用于以下变更类型：

- 新增或修改接口
- 新增或修改数据库结构
- 新增或修改事件格式
- 新增或修改业务状态与流程
- 新增或修改三方系统映射和回调
- 新增或修改前端配置驱动结构
- 新增或修改权限规则
- 新增或修改跨模块公共模型

## 4. 标准执行流程

任何中大型变更，默认按照以下流程执行。

### 第 1 步：确认变更意图

先明确本次变更的业务目标：

- 改的是结构，还是表现？
- 改的是规则，还是文案？
- 改的是边界契约，还是内部实现？
- 改动是新增能力，还是修复已有逻辑？

输出要求：

- 能用一句话描述本次变更真正影响的业务事实。

如果无法清楚描述，说明需求还没有被正确抽象。

### 第 2 步：识别源头定义

在动手改代码前，必须先识别本次变更的“源头定义”。

常见源头包括：

- API contract
- GraphQL schema
- Proto definition
- database schema / migration
- domain model
- state machine
- event schema
- permission model
- adapter contract
- component spec / form schema

输出要求：

- 明确记录“本次变更的源头是什么”。

禁止行为：

- 直接改页面展示。
- 直接改 controller / handler。
- 直接改 mapper / serializer。
- 直接改 consumer / callback handler。

如果源头未变，下游改动原则上不应先开始。

### 第 3 步：列出影响面

识别完源头后，必须显式列出受影响模块。

至少检查以下维度：

- 前端展示层
- 前端表单或配置层
- 前端请求层和类型层
- 后端接口层
- 后端领域层
- 数据库存储层
- 事件生产者和消费者
- 定时任务和补偿逻辑
- 三方适配层
- 监控、告警、日志、报表

输出要求：

- 形成一份“影响面清单”。

原则：

- 不允许靠记忆认为“应该只有这里受影响”。
- 必须显式排查低频路径和边缘路径。

### 第 4 步：先修改源头

先改源头定义，再改其他模块。

示例：

- 改接口时，先改 OpenAPI / GraphQL / Proto。
- 改表结构时，先改 migration / schema。
- 改状态流转时，先改状态机或领域规则。
- 改事件时，先改 event schema。
- 改组件能力时，先改 component spec。
- 改三方映射规则时，先改 adapter contract。

输出要求：

- 源头定义必须先于消费层提交。

如果系统支持自动派生：

- 源头修改后应立刻生成相关类型、SDK、映射器、校验器或文档。

### 第 5 步：更新派生产物

所有从源头派生的内容必须同步更新。

典型包括：

- generated types
- SDK
- validators
- form metadata
- panel config
- registry
- mock contracts
- DTO / mapper
- migration helpers

输出要求：

- 所有派生产物与源头保持一致。

禁止行为：

- 源头已改，但保留过期生成结果。
- 因为“暂时能跑”而跳过派生更新。

### 第 6 步：补齐消费层实现

在源头和派生产物更新后，再逐项处理消费层。

典型消费层包括：

- 页面组件
- 表单和配置面板
- 前端调用逻辑
- controller / resolver / handler
- domain service
- repository / query layer
- producer / consumer
- adapter / callback handler

执行要求：

- 按影响面清单逐项确认。
- 不允许只修主路径，不检查边缘路径。
- 不允许只修报错点，不补齐同源相关逻辑。

### 第 7 步：补齐验证

所有较大变更都应补齐对应验证，而不是只让代码“编译通过”。

至少检查：

- type check
- lint
- unit test
- contract test
- integration test
- migration test
- sandbox / replay test
- smoke test

验证要求：

- 改契约必须验证契约。
- 改结构必须验证结构兼容性。
- 改流程必须验证关键状态流转。
- 改三方对接必须验证边界行为。

### 第 8 步：发布前检查

对高风险变更，发布前必须做额外确认。

至少检查：

- 是否有 migration 风险。
- 是否有三方协议变更风险。
- 是否有历史数据兼容风险。
- 是否有异步重试、重复消费、乱序问题。
- 是否具备 feature flag、灰度和回滚方案。
- 是否具备关键监控和告警。

输出要求：

- 没有发布策略的高风险变更，不应直接上线。

## 5. AI Coding 场景专用要求

当使用 AI 参与开发时，本规范应额外强调以下要求：

### 5.1 不允许只根据当前打开文件修改

AI 不应只围绕当前文件做补丁式修改，而应先识别：

- 源头定义在哪里。
- 受影响的边界有哪些。
- 哪些派生产物需要更新。
- 哪些验证必须补齐。

### 5.2 不允许把“编译通过”视为完成

AI 的任务完成标准不应仅为：

- 无语法错误
- 无类型报错
- 页面能打开

真正完成标准应包括：

- 源头已更新
- 相关派生产物已同步
- 相关消费层已补齐
- 必要验证已通过

### 5.3 不允许只修当前报错点

AI 不应因为某处报错被修复，就默认任务完成。

必须继续检查：

- 同类消费者是否同步修改
- 隐式依赖是否同步修改
- 历史数据或边缘路径是否兼容
- 契约测试和集成测试是否补齐

### 5.4 要求显式说明修改路径

对于中大型变更，AI 应在执行前或执行中能说明：

- 这次改动的源头是什么
- 影响面有哪些
- 计划修改哪些层
- 计划执行哪些验证

如果无法说明，则说明 AI 对任务尚未建立完整结构认知。

## 6. 分场景执行指引

### 6.1 接口变更

执行顺序：

1. 修改 API contract。
2. 生成或更新类型、SDK、mock。
3. 更新服务端实现。
4. 更新前端消费层。
5. 运行契约测试和集成测试。

禁止事项：

- 先改前端请求结构，再补接口定义。
- 接口字段已变，但 SDK 或类型未更新。

### 6.2 数据库结构变更

执行顺序：

1. 修改 schema / migration。
2. 更新 ORM 类型、查询映射、校验逻辑。
3. 更新服务端领域逻辑和接口层。
4. 验证历史数据兼容。
5. 运行 migration test 和关键回归测试。

禁止事项：

- 先改查询逻辑，再补 migration。
- 只改新数据路径，不验证历史数据。

### 6.3 业务规则或状态流转变更

执行顺序：

1. 修改状态机、领域规则或聚合模型。
2. 更新入口层调用。
3. 更新页面按钮、展示逻辑、任务系统、消息处理器。
4. 增加关键状态流转测试。

禁止事项：

- 在 controller、页面、消费者里分别补规则。
- 主链路已改，但异步或补偿链路未改。

### 6.4 三方系统对接变更

执行顺序：

1. 修改 adapter contract。
2. 更新字段映射、错误码映射、回调解析规则。
3. 更新内部领域模型适配层。
4. 运行 sandbox / mock / replay 验证。
5. 确认幂等、重试、熔断、告警和回滚方案。

禁止事项：

- 让三方字段直接渗透到内部多个模块。
- 只在一个 handler 中补丁式兼容。

### 6.5 配置驱动或低代码场景变更

执行顺序：

1. 修改 schema / component spec。
2. 更新默认值、校验、panel metadata、registry。
3. 更新渲染层和导出层。
4. 运行 contract test 和关键渲染验证。

禁止事项：

- 只改配置表，不改渲染层。
- 只改渲染细节，不回到 spec 源头收敛。

## 7. 完成标准

一次变更只有满足以下条件，才可视为完成：

- 已识别并修改源头定义。
- 已列出并检查主要影响面。
- 已更新派生产物。
- 已补齐受影响消费层。
- 已完成必要验证。
- 对高风险变更已完成发布前检查。

以下情况不能视为完成：

- 只是当前报错消失。
- 只是编译通过。
- 只是主链路看起来能运行。
- 只是改动文件本身没有问题。

## 8. 评审要求

代码评审时，默认检查以下问题：

- 这次变更的源头定义是否被正确识别？
- 是否存在绕开源头直接修改下游的情况？
- 是否存在双写结构没有收敛？
- 是否列出了关键影响面？
- 是否补齐了契约或集成验证？
- 是否遗漏了数据库、异步链路或三方边界？

如果这些问题回答不清楚，即使代码表面可运行，也不应直接通过评审。

## 9. 可作为团队约定的简版规则

如果需要形成简洁版本，可直接采用以下规则：

1. 改任何需求，先找源头定义。
2. 源头不改，不先动消费层。
3. 改完源头，先更新派生，再改实现。
4. 改完实现，必须跑对应验证。
5. 只修报错点不算完成，补齐全链路才算完成。

## 10. 结语

这份执行规范的意义，在于把“尽量别漏改”变成“必须按路径完成修改”。

它不依赖人的记忆力，也不依赖 AI 的自觉性，而是通过统一执行顺序，把变更过程收敛为一条可检查、可复用、可自动化的路径。

长期来看，这会显著降低复杂系统在持续迭代中的隐性一致性风险。
